## Learned User Preferences

- When changing this Raycast GitLab extension, preserve existing code style and structure; do not refactor unrelated patterns or drift from local conventions.
- Run requested terminal, lint, and Raycast dev/debug commands yourself when asked instead of only describing them.
- Communicate in Russian in chat; code comments, UI copy, and docs follow existing project language.
- Never commit or push without explicit user request.
- Refactoring must preserve current behavior unless the user explicitly requests a behavior change.
- Data fetching: prefer `useCachedPromise`, `useCachedState`, and `Cache` from `@raycast/utils` over custom cache hooks; prefer `usePromise` over `useState`+`useEffect` when async fetch is the only state update path; handle errors inline or via hook `onError` (not toast-only `useEffect`). When the call site shows errors, pass `onError: () => undefined` to suppress default utility toasts.
- React style: declare JSX inline—do not store elements in `useMemo`; PascalCase for functions that return components; inline 1–3 line single-use expressions (except enum/mapping tables); omit redundant guards when types/API contracts already guarantee values.
- Naming: prefer short or full names for parameters and variables over one-letter names (including closure args)—e.g. `branch`/`repo` or `currentBranch`/`selectedRepository`, not `s`/`m`/`b` or opaque abbreviations like `brch`/`rp`; don't over-lengthen names when local context makes a shorter name sufficient.
- Dates: `formatDate` for relative display and `formatDateTime` for absolute strings (tooltips, etc.) in `utils.ts`; avoid ad-hoc `toLocaleString` or legacy date helpers.

## Learned Workspace Facts

- Search Merge Requests: UI in `src/components/mr_search.tsx` (entry `src/mr_search.tsx`); AI tool `src/tools/search-merge-requests.ts` is separate. `MyProjectsDropdown` with `includeAllItem={false}` (project required before load); sections "Created by me", "Assigned to Review" (hidden when empty), "Other"; Sort By (`mr_sort.tsx`, ⌘S) sets GitLab `order_by` ("Default" omits param). Search MR prefs use `useCachedState`; fetch uses `usePaginatedMergeRequests`.
- `MyProjectsDropdown` in `src/components/project.tsx` supports `includeAllItem` (default `true`); My Merge Requests and Reviews keep "All Projects". My MR fetch (`useMyMergeRequests` in `mr_my.tsx`) uses `usePaginatedMergeRequests`; cache key excludes project—filter stays client-side in `MyMergeRequests`.
- My Merge Requests hides author subtitle and avatar on list rows via `showAuthor={false}` on `MRListItem`; Search MR and other MR lists keep the default (`showAuthor` true). MR list row subtitle is hidden while the side panel is open.
- MR list side panel: `useMRListDetails()` + `MRListDetailsToggleAction` (⌘⇧D). Metadata column: `useMRListMetadata()` + `MRListMetadataToggleAction` (⌘⇧I, only when side panel open). Side panel uses `MRListDetailMetadata`, full detail uses `MRDetailMetadata` in `mr_metadata.tsx`; the former `listdetails` preference was removed from `package.json`.
- Shared MR UI in `mr.tsx`, `mr_actions.tsx`, `mr_metadata.tsx`, `mr_discussions.ts`, `mr_status.ts`. Do not fetch MR discussions per list row; `useMRDiscussionStats` (`useCachedPromise` in `mr_discussions.ts`) is for detail/side panel only. Metadata label is "Discussions" (`resolved/resolvable` only; not `user_notes_count`); no discussion count in list accessories. MR list Copy section (`MRCopySection` in `mr_actions.tsx`): Copy URL (⌘.), Copy Markdown (⌘⌃.) on list rows.
- My Merge Requests and Reviews use GitLab API default order (`created_at` desc); no client-side re-sort.
- GitLab [Merge requests API](https://docs.gitlab.com/api/merge_requests/): list endpoints may return stale `merge_status` / `has_conflicts` unless `with_merge_status_recheck=true` (async recheck, not guaranteed; may be ignored on 15.11+ without Developer+ if `restrict_merge_status_recheck` is enabled). `with_merge_status_recheck` does not control `head_pipeline`. Avoid `view=simple` when full MR fields are needed. Extension defaults in `getMergeRequests` / `getGroupMergeRequests`: `with_labels_details=true` and `with_merge_status_recheck=true` (callers can override).
- `MRListItem` must not call GitLab per row for approvals, full MR, or discussions—only `useMRPipelines(mr, { limit: 1 })` when list payload has no `head_pipeline` (and not when `has_conflicts`). Pipeline accessory: prefer `head_pipeline` from list (`getMRHeadPipelineStatus`, `parseHeadPipelineFromJson`); else `useMRPipelines(mr, { limit: 1 })`; `getCIJobStatusIcon`; hidden when `has_conflicts`. Full MR pipelines use `useMRPipelines(mr, { all: true })` in `MRPipelineList`. Discussions/full MR/approvals fetches belong in side panel (`MRListDetail`) or detail (`MRDetail`) only.
